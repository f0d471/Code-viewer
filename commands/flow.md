# /code-viewer flow — 信号生命周期追踪

## 定位

flow 不是"看看数据怎么流"，而是**追踪一条信号/数据的完整生命周期**——从触发到终结，每一步变换不跳步，用时序图表示并行。

和 scan / deep 的区别：
- scan = 全局地图（所有子系统）
- deep = 深挖一个模块（WHY/WHAT/HOW + 逐行）
- flow = 追踪一条线（跨越多个模块的信号生命周期）

---

## 触发条件

用户调用 `/code-viewer flow [功能名或信号名]`。

示例：
- 软件：`/code-viewer flow 用户登录`、`/code-viewer flow HTTP 请求处理`
- 硬件：`/code-viewer flow 一条指令的执行`、`/code-viewer flow 中断响应`、`/code-viewer flow AXI4 读事务`

---

## 流程

### Step 1：定义信号边界

明确追踪的信号是什么：

**场景类型检测**：根据项目类型和用户描述，自动选择术语体系和图形表达方式：

| 类型 | 信号叫法 | 时间尺度 | 并行表示 | 终止条件 |
|---|---|---|---|---|
| **硬件**（Verilog/SV/Chisel） | 信号 / 中断 / 事务 | 时钟拍数 | 时序图（泳道+时间轴） | 写回寄存器 / 输出端口 |
| **软件后端**（HTTP/API） | 请求 / 事件 / 消息 | 毫秒 / 异步等待 | 调用序列图 | 返回响应 / 写入 DB |
| **软件前端**（UI/事件） | 事件 / 操作 / 状态变更 | 事件循环 tick | 事件传播图 | DOM 更新 / 状态稳定 |
| **系统**（内核/驱动） | 中断 / 信号 / 系统调用 | 纳秒~毫秒 | 上下文切换图 | 返回用户态 |

检测依据：
- 文件扩展名（`.v`/`.sv`/`.chisel` → 硬件；`.py`/`.js`/`.go` → 软件）
- 用户描述的关键词（"中断""流水线""AXI" → 硬件；"登录""HTTP""请求" → 软件）
- 项目配置文件（`Makefile` 含 Verilator → 硬件；`package.json` → 软件）

后续所有术语按检测结果切换，不再混用。

1. **信号名**：这个信号/数据叫什么？（一条指令、一个 HTTP 请求、一个中断信号、一笔 DMA 传输）
2. **起点**：信号从哪里产生？（入口点、外设引脚、定时器触发）
3. **终点**：信号在哪里终结？（写回寄存器、返回响应、触发下一级）
4. **时间尺度**：这条信号的生命周期是几拍（硬件）、几毫秒（软件）、还是几次函数调用？

### Step 2：逐跳追踪

从起点开始，追踪信号经过的每一个模块/阶段：

每个节点写清楚：
- **输入**：信号到达这个节点时长什么样？（数据格式、控制信号、请求参数）
- **变换**：这个节点对信号做了什么？（计算、转发、缓存、转换协议）
- **输出**：信号离开这个节点时变成什么样？
- **耗时**：这个节点停留多久？（硬件：几拍；软件：同步/异步）

**不跳步**。从入口到终点，每一步都要写。如果 A 调用 B 调用 C，不能跳成 A → C。

### Step 3：识别拓扑结构

信号流不只是线性的。识别以下结构：

| 结构 | 描述 | 图形表示 |
|---|---|---|
| **串行** | A 完成后 B 才开始 | `A → B → C` |
| **分支** | 条件判断走向不同路径 | 分叉节点（橙色） |
| **并行** | 多个阶段同时进行 | 时序图（泳道 + 时间轴） |
| **汇聚** | 多路信号合并为一路 | 汇合节点 |
| **回环** | 信号循环（流水线反馈、重试） | 环形箭头 |

硬件项目中，并行（pipeline stages、AXI4 的 5 通道）是最常见的结构，必须用时序图表达。

### Step 4：时序图（并行结构）

当信号在多个模块间并行流动时，用 SVG 时序图：

**时序图规范**：
- 横轴 = 时间（拍/周期），纵轴 = 模块/阶段
- 每个模块一行泳道
- 箭头 = 信号传递（从源模块到目标模块）
- 虚线 = 有效信号持续时间
- 标注 = 信号名 + 值

**硬件时序图示例**（5 级流水线一条指令的生命周期）：

```
        T0    T1    T2    T3    T4    T5
  IF  : |取指|
  ID  :       |解码|
  EX  :             |执行|
  MEM :                   |访存|
  WB  :                         |写回|
```

**软件时序图示例**（HTTP 请求生命周期）：

```
        T0      T1       T2       T3       T4
 Client: |请求|
 Gateway:      |路由|
 Handler:             |业务逻辑|
 Database:                    |查询|
 Response:                           |返回|
```

### Step 5：输出报告

1. **Read `templates/shared-styles.css`**，将完整内容注入报告 `<style>` 标签。
2. 按 `templates/flow-report.html` 格式输出自包含 HTML：

1. **信号定义**（信号名、起点、终点、时间尺度）
2. **入口点**（callout-entry）
3. **数据流图**（水平 flow-container + 完整 SVG）
4. **时序图**（并行结构的 SVG 时序图）
5. **关键节点详情**（每个节点：代码 + 输入/变换/输出表）
6. **外部依赖**
7. **建议下一步**

分批规则：§1-§3 为一批，§4-§5 为一批（节点多时每批 2-3 个节点），§6-§7 为一批。

---

## 质量标准

- 读完报告，能回答：这个信号从哪来、经过了哪些模块、每一步变成什么样、最终到哪去
- 术语体系与项目类型匹配（硬件用"信号/拍"，软件用"请求/毫秒"，前端用"事件/tick"）
- 并行结构用时序图表达，不能只用文字说"同时进行"
- 每个节点的变换必须写清楚输入→输出的具体变化，不能写"经过处理"
- 硬件信号要写明在哪个时钟沿生效、持续几拍
- 软件请求要写明同步/异步、是否有阻塞等待
