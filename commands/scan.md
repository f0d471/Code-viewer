# /code-viewer scan — 项目架构 Wiki

## 触发条件

用户调用 `/code-viewer scan [路径]`，或直接 `/code-viewer` 无其他关键词。

## 定位

scan 不是"快速鸟瞰"，而是生成一份**项目架构 Wiki**——帮用户建立完整的心智模型。
输出质量标准：用户读完能画出项目架构图，能说出每个子系统干什么。

---

## 流程

### Step 1：生成深度文件树

对目标路径执行 tree，关键目录展开到底。

- **关键目录**（必须展开到底）：`src/`、`lib/`、`cmd/`、`app/`、`vsrc/`、`csrc/`、`include/`、`core/`、`rtl/`、`sim/`、`test/`
- **大目录**（>200 文件，只展两层 + 标注省略）：`vendor/`、`third_party/`、`node_modules/`、`IP/`、`ip/`
- **自动生成目录**（跳过）：`.git/`、`build/`、`dist/`、`target/`、`__pycache__/`
- 标注每个目录的文件数，帮助用户判断重要性

### Step 2：子系统分解

不要只看技术栈配置文件。通读目录结构 + README + 关键文件，识别项目的**子系统层**。

分解方法：
1. 看顶层目录——每个一级目录大概率是一个子系统
2. 读各子系统的 README / Makefile / 入口文件，确认职责
3. 识别子系统之间的依赖关系（谁依赖谁、数据怎么流）
4. 判断项目的"分层模型"——常见的有：
   - 硬件项目：硬件核心 → 仿真层 → 运行时 → 应用
   - Web 项目：前端 → 后端 → 数据库
   - 编译器项目：前端 → 中端 → 后端
   - 系统项目：内核 → 驱动 → 用户态

输出：
- 子系统清单（名称、路径、一句话职责）
- 分层模型（谁在上、谁在下、谁平行）
- 依赖关系图（SVG）

### Step 3：定位每个子系统的最小核心

对每个子系统，独立定位它的入口点。

定位方法（按优先级）：
1. 看构建配置的 entry 字段 / main class / top module
2. 搜 `main`、`index`、`app`、`server`、`module top`、`module tb_` 等关键词
3. 看 README 里有没有启动说明
4. 看 Dockerfile 的 `CMD` / `ENTRYPOINT`
5. 硬件项目：看 testbench 实例化的模块 / 仿真脚本的 `TOP_MODULE`

每个子系统输出：
- 入口文件路径
- 入口代码片段（不超过 30 行）
- 它做了什么（一句话）
- 它直接调用了哪些模块

### Step 4：架构全景图

**先给用户一张总图，满足 VSL 原则的"全局优先"。**

画一张 SVG 架构图，展示所有子系统的关系：
- 垂直布局：底层在下、上层在上
- 连线表示依赖关系（箭头从依赖者指向被依赖者）
- 每个节点标注子系统名 + 一句话职责
- 用洋葱模型色区分层级（L0 核心 → L3 外围）

### Step 5：子系统详细分析

对每个子系统，做一次 mini-scan：

| 项目 | 内容 |
|---|---|
| 职责 | 这个子系统在整个项目中的角色 |
| 技术栈 | 它用什么语言/框架/工具写的 |
| 文件规模 | 多少文件、大约多少行 |
| 值得读指标 | 入口文件行数（越短越值得先读）、核心模块数量、被几个子系统依赖 |
| 目录结构 | 关键目录的用途 |
| 入口 | 最小核心（Step 3 的结果） |
| 公开接口 | 它对外暴露了什么（API / 端口 / 函数） |
| 依赖 | 它依赖哪些其他子系统 |
| 被谁依赖 | 哪些子系统依赖它 |

### Step 6：输出报告

1. **Read `templates/shared-styles.css`**，将完整内容注入报告 `<style>` 标签。这是 CSS 唯一来源，不能跳过或凭记忆写。
2. 按 `templates/scan-report.html` 格式输出自包含 HTML。包含：

1. **项目概览**（一句话 + 技术栈 + 文件规模）
2. **深度文件树**（关键目录展开到底）
3. **子系统分解**（清单 + 分层模型 + 依赖关系图）
4. **推荐阅读顺序**（基于洋葱模型 L0→L3，标注每个步骤读什么文件、理解什么）
5. **架构全景图**（SVG）
6. **你的目标是什么？**（目标驱动选择：理解功能→flow，搞清组织→scan→deep，修改模块→deep，随便问问→ask）
7. **子系统详细分析**（每个子系统一个 section，用 `<details>` 折叠，含"值得读"指标）
8. **推荐下一步**（哪个子系统最值得先深入）

报告分批生成，单次不超 3 个 section（见 `core/rules.md` 规则 12）。
