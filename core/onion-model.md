# 洋葱模型

> **适用范围**：deep 命令（模块解剖）中分析模块在项目中的层级定位时使用。scan 命令使用子系统分解（见 `commands/scan.md`），不使用洋葱模型。

## 分层策略

由内向外，每层只负责一种抽象级别：

### 第 0 层：最小核心骨架

程序的入口点 + 主循环/调度器。这是整个项目的"心脏"。

特征：
- 通常是项目中**最短**但**最重要**的文件
- 不包含具体业务逻辑，只做"组装"和"启动"
- 找到它 = 找到理解整个项目的支点

典型形式：
- 主函数 + 事件循环
- 路由注册 + 服务器启动
- 组件树的根节点
- 命令行解析器 + dispatch
- Verilog 顶层模块（`module top`）+ 子模块实例化

### 第 1 层：核心数据流

从入口点直接调用的模块。它们定义了"数据怎么进来、怎么处理、怎么出去"。

特征：
- 被入口点直接 import / require / include
- 通常是 controller、handler、router 级别
- 数量不多（3-10 个），但每个都很关键

### 第 2 层：业务逻辑模块

被第 1 层调用的模块，包含实际的业务规则。

特征：
- 不直接接触外部（网络、文件、用户输入）
- 通过第 1 层接收数据，处理后返回
- 通常可以单独测试

### 第 3 层：工具层

被多个层级共同依赖的底层模块。

特征：
- 配置、日志、数据库连接、工具函数
- 通常被很多文件 import
- 改动频率低，但影响面广

## 使用方式

在 deep 命令中，用洋葱模型判断目标模块属于哪一层（L0-L3），帮助理解它在项目中的角色。依赖关系在 deep 中完全展开（不限于一层），用 SVG 呈现调用关系图。
