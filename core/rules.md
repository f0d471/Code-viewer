# 硬性规则

agent 分析代码时必须遵守以下规则。任何一条违反都是失格。

---

## 内容规则

1. **不编造理解。** 没有读过的文件、没有 grep 过的调用关系，不能出现在报告里。宁可标"待深入"，不能编造"这个模块负责 xxx"。
2. **调用链不跳步。** 入口 → 第一层 → 第二层，每一步都要写清楚。不能从入口直接跳到叶子节点说"最终调用了 xxx"。
3. **每个模块说清楚三件事。** 做什么（一句话职责）、谁调用了我（向上依赖）、我调用了谁（向下依赖）。缺任何一项都不算分析过这个模块。
4. **代码引用必须带文件路径和行号。** 不能说"在主函数里调用了 xxx"，要说"`src/main.py:42` 调用了 `config.load()`"。
5. **入口必须先找到。** 没有定位到入口就开始分析模块，等于没有地图走进迷宫。
6. **不省略"无聊"的模块。** 配置、工具函数、常量定义——这些被多个模块共同依赖，跳过它们会导致理解断层。
7. **待深入要诚实标记。** 用 `callout-warning` 标记尚未展开的模块，不能假装已经分析过。
8. **技术栈判断必须有依据。** 不能猜"这大概是个 Python 项目"——要列出你看到了什么文件（`requirements.txt`、`pyproject.toml`）。
9. **tree 输出要截断。** 大项目（>200 文件）只展示核心目录，标注"省略 xx 个文件"，不要刷屏。
10. **数据流追踪必须写变换。** 每一步都要写：输入是什么 → 做了什么变换 → 输出是什么。不能只写"经过模块 A 处理"。

---

## 执行规则

11. **每次只分析一层。** scan 只管全局 + 子系统分解 + 架构全景图 + 推荐阅读顺序。deep 根据传入路径自动判断模式——模块模式（Phase 1 完全展开 + Phase 2 热度筛选 + 逐行解读）或子系统模式（模块清单 + 调用关系 + 数据流 + 边界）。flow 只追踪一条数据流。ask 回答一个问题即止，不展开系统性分析。不一次全干完。
12. **分批生成报告，单次不超 3 个 section。** HTML 报告按 section 填充，每次最多写 3 个 section 的内容，写完暂停等用户确认再继续。避免上下文爆满导致质量塌方。
    - scan 报告推荐分批：§1 概览 + §2 文件树 → §3 子系统分解 + §4 架构全景图 → §5 子系统详细分析 + §6 推荐下一步
    - deep 报告推荐分批：模块模式—Phase 1 §1 动机 + §2 功能 + §3 核心实现 → Phase 1 §4 依赖 + §5 内部结构图 → Phase 2 §6 热度筛选 → Phase 2 §7 逐行解读（按文件分批，每批 1-2 个核心文件）→ §8 下一步；子系统模式—§1 模块清单 + §2 调用关系 → §3 数据流 + §4 模块边界 → §5 报告输出
    - flow 报告推荐分批：§1 信号定义 + §2 入口 + §3 数据流图 → §4 时序图 + §5 节点详情（每批 2-3 个节点）→ §6 依赖 + §7 下一步
13. **每次暂停时告诉用户下一步做什么。** 不要让用户猜——明确说"§1–§2 已完成，确认后我继续写 §3 模块表"或"可以用 `/code-viewer deep xxx` 深入这个模块"。
14. **每次重新读。** agent 不凭记忆，每次命令触发时重新 Read 对应的 command 文件和 core 文件。每个 section 开始前重新 Read 该 section 依赖的 renderer/template 文件。

## Forbidden 列表（强制禁止）

### 文风层

| 禁止 | 替代 |
|---|---|
| "大致负责 xxx"、"可能用于 xxx" | 读代码后写确切职责，读不了就标"待深入" |
| "结构清晰"、"代码质量不错"、"设计良好" | 删掉，这是废话 |
| "显然"、"不难看出"、"简单来说" | 老老实实写调用关系和代码引用 |
| "让我们一起看看"煽动开场 | 直接开始分析 |
| 一段话里没有文件路径 | 加上 `path/to/file:line` |
| 装饰性 emoji | emoji 只用在清单标记和 callout 标题 |
| 用术语解释术语（"handler 处理请求"） | 写清楚：handler 接收什么参数、调用什么函数、返回什么 |

### 视觉层

| 禁止 | 替代 |
|---|---|
| ASCII art 画调用关系图 | 用 SVG（见 `renderers/callgraph.md`） |
| 所有 callout 用同一个颜色 | 按语义选色：入口紫、依赖蓝、警告红、提示橙 |
| 代码块不标语言 | 加 `language-xxx` class 让 highlight.js 高亮 |
| 长代码块不截断 | 只引关键行，标注文件路径和行号范围 |

### 内容层

| 禁止 | 替代 |
|---|---|
| 说"负责业务逻辑"不说具体是什么逻辑 | 写清楚：接收什么数据、做什么计算、返回什么结果 |
| 列模块表但不标层级 | scan 用 layer-badge（L0-L3）标子系统层级，deep 用 code-block 标文件层级 |
| 画调用图但跳过中间层 | 入口 → A → B → C，不能跳成 入口 → C |
| 只给函数名不给调用关系 | 每个函数标"被谁调用"和"调用了谁" |
| 把 import 列表当分析结果 | import 是起点不是结论，要追踪 import 后实际用了什么 |
| deep 报告跳过 WHY 直接讲 HOW | 每个代码块先回答"为什么存在"，再讲"是什么""怎么实现" |
| deep Phase 2 对所有代码块强制完整三问 | 先做热度筛选，只对核心逻辑做 WHY/WHAT/HOW；核心块太多时只覆盖最重要的 ~60%，其余标记"可选深入" |
| deep 跳过 Step 0 预检索直接读代码 | 先 grep 谁 import 了它、谁调用了它，建立客观调用关系后再解读 |
| deep Slop Test 能否重写核心逻辑 | 不要求能重写每一行，但要求能重写核心逻辑块 + 知道胶水代码的位置和一句话作用 |
| deep 子系统模式跳过模块边界 | 每个模块必须标清：对外接口、内部不暴露、依赖边界 |

---

## Slop Test（AI 味检测）

写完报告后自问：

- **scan**：读完报告，能不能画出项目架构图、说出每个子系统干什么？不能 = 不达标。
- **deep**：读完报告，能不能从零重写这个模块的核心逻辑？知不知道胶水代码在哪、干什么？（子系统模式：能不能说出每个文件干什么、模块间怎么协作、边界在哪？）不能 = 不达标。
- **flow**：读完报告，能不能回答这个信号从哪来、每一步变成什么、最终到哪去？不能 = 不达标。
- 如果删掉所有 callout 和图表，剩下的文字还能不能独立描述结构？不能 = 对视觉辅助依赖过重。
- 一个熟悉这个项目的开发者看到报告，会不会觉得"这是 AI 没读代码就编的"？会 = 分析偷懒了。

---
