# VSL 设计原则

本 skill 的用户是视觉-空间型学习者（Visual-Spatial Learner）。Linda Silverman 的研究表明：VSL 用画面而非文字思考，整体把握而非线性推进，一旦在脑中建成正确的视觉模型，理解就是永久的。

代码阅读同样适用——一个在脑中建成完整调用关系图的人，比只读过源码的人理解更深、记得更久。

以下原则渗透到输出的每个环节：

---

## 1. 全局优先

先看整张地图，再走某条路。每次分析前先给出项目文件树（tree），让用户在读任何代码之前就知道项目多大、分几层、入口在哪。

**code-viewer 对应命令**：`/code-viewer scan` 的核心就是这一条——文件树 → 子系统分解 → 架构全景图，全部是全局视野。

## 2. 图即内容

图是理解的载体，文字是对图的批注。每个模块关系、调用链、数据流都有对应的 SVG。不要把图当作"装饰"——如果删掉图之后文字仍然完整，说明这个图画错了。

**code-viewer 的图**：
- 模块依赖树：谁包含谁、谁依赖谁
- 调用关系图：谁调用了谁、调用链多深
- 数据流图：数据从哪来、经过哪些变换、到哪去

## 3. 关系优先于定义

VSL 通过"这个东西和已知的东西怎么连"来理解新模块。先看谁调用谁、谁依赖谁，再看函数签名和类型定义。孤立地读一个文件对 VSL 效率最低。

**实际操作**：
- 读模块前先 grep 谁 import 了它
- 读函数前先看它在调用链中的位置
- 读数据结构前先看它被哪些模块使用

## 4. 模式 > 列表

结构化的视觉分组比平铺的文件列表更易记。用子系统分层呈现（层级 badge + 架构图），比列一个 50 行的文件清单有效得多。

**具体做法**：
- scan：子系统表格用 layer-badge 标层级（L0-L3），SVG 架构图展示依赖
- deep：模块内部用 code-block 组件（WHY/WHAT/HOW），SVG 展示数据通路
- 调用关系用 SVG 树形图，不用文字描述
- 对比两个模块时用并列卡片，不用段落

## 5. 一次理解 = 永久记忆

调用链不能跳步。VSL 在脑中构建画面，跳一步画面就断了，断了就丢了。宁可多写 3 行代码引用，也不要用一句"这里调用了 xxx"糊弄过去。

**强制规则**：
- 数据流追踪时，每一步都要写：输入是什么 → 做了什么变换 → 输出是什么
- 调用关系图中，不能省略中间层（不能从入口直接跳到叶子节点）
- 标记"待深入"的模块要诚实，不编造理解
